# 多轮迭代指南

## 何时启动新的 `sh-start` vs 继续用 `sh-pack`

| 场景 | 动作 | 原因 |
|------|------|------|
| 目标不变，下一批工作 | `sh-pack` | 目标未变，只需定下一轮的范围 |
| 从 v1 中学到东西后目标部分调整 | `sh-pack` 并更新引用的主文档 | 主方向成立，细节演进 |
| 目标根本性改变，或切换到不同方案 | `sh-start` | 需要重新头脑风暴、路线对比和决策记录 |
| 重大技术栈或架构变更 | `sh-start` | 之前的路线和权衡不再适用 |

## 归档历史文档

完成一轮后：

1. **执行包** — 保留在 `docs/plans/` 中，带日期前缀和 `Completed` 状态标记
2. **主文档** — 如果被新的 `sh-start` 取代，用版本后缀重命名（如 `main-v1.md`）并标记为 `Archived`
3. **学习摘要** — 始终保留；它们是项目的永久知识

## 追踪轮次间的依赖

每个 `sh-pack` 应在"依赖的上游结论 / 主文档"字段中引用其上游：

- 第 1 轮：引用 `sh-start` 决策记录
- 第 2 轮及以后：引用上一轮的执行报告 + 任何更新过的主文档
- `sh-recover` 之后：引用恢复输出作为新的上游

这形成一条线性链：`sh-start → pack-1 → run-1 → pack-2 → run-2 → ...`

如果在链中间发生 `sh-recover`，则分叉：`... → run-N → recover → pack-N+1 → ...`
