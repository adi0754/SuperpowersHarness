# 端到端示例：给博客加 AI 摘要生成

> 这个示例用一个虚构的小功能，完整演示 sh-start → sh-pack → sh-run 三个入口的输入和输出。

---

## 背景

你有一个技术博客系统（Next.js + PostgreSQL），目前支持 Markdown 写作、标签分类、RSS 输出。
现在想加一个功能：**文章发布后自动生成一段 AI 摘要，展示在文章列表和 SEO meta 中**。

---

## 第一步：`/sh-start` — 定方向

### 你的输入

```
/sh-start

- 当前项目 / 功能背景：技术博客，Next.js + PostgreSQL，已有 Markdown 写作、标签分类、RSS
- 这次想解决的问题：文章发布后自动生成 AI 摘要，用于列表预览和 SEO meta description
- 为什么现在做：手写摘要太耗时，经常偷懒不写导致列表页和 SEO 效果差
- 已知约束：不做全文改写、不做多语言摘要、API 成本要可控
```

### Skill 给你的输出（摘要）

**路线比较：**

| 路线 | 方案 | 优点 | 缺点 | 对 MVP 影响 |
| ------ | ------ | ------ | ------ | ------------- |
| A | 发布时预生成：文章保存时调 LLM，摘要写入数据库 | 读取零延迟，离线可用，成本可预测 | 修改文章需重新生成；发布时多等几秒 | MVP 最小 |
| B | 请求时实时生成：用户访问时调 LLM，结果缓存 | 永远是最新内容；不改发布流程 | 首次访问慢 2-3 秒；缓存失效策略复杂；并发下可能重复调用 | 中等复杂 |
| C | 混合方案：发布时预生成 + 内容变更时异步更新 | 兼顾速度和新鲜度 | 需要消息队列或后台任务；MVP 阶段过度设计 | 超出 MVP |

**Skill 推荐路线 A，你拍板确认。**

**你的决策记录：**

- 一句话目标：文章发布时自动调 LLM 生成摘要，存入数据库，列表页和 SEO 直接读取
- 当前 MVP：发布时生成一次，不做自动更新
- 明确不做：实时生成、多语言摘要、全文改写、摘要风格自定义
- 选择路线：A — 发布时预生成
- 不选其他路线的理由：B 首次访问延迟高且缓存策略复杂，C 引入异步任务超出 MVP 范围

---

## 第二步：`/sh-pack` — 压执行包

### 你的输入

```
/sh-pack

- 当前主目标：完成 AI 摘要生成 MVP
- 当前这一轮要完成的结果：文章发布 → 调 LLM 生成摘要 → 存入数据库 → 列表页展示 → SEO meta 输出
- 当前明确不做：摘要自动更新、多语言、风格自定义、批量补生成历史文章
- 依赖的上游结论 / 主文档：sh-start 决策记录（路线 A — 发布时预生成）
```

### Skill 给你的执行包（摘要）

**目标结果：** 文章发布后数据库中自动出现 AI 摘要，列表页和 SEO meta 直接使用。

**任务列表（5 项）：**

1. 数据库：`posts` 表新增 `ai_summary` 字段（text, nullable）
2. 后端：封装 LLM 摘要生成服务（输入 Markdown 正文，输出 ≤150 字摘要）
3. 后端：在文章发布/更新 API 中调用摘要服务，写入 `ai_summary`
4. 前端：文章列表页优先展示 `ai_summary`，无摘要时回退到正文截断
5. SEO：`<head>` 中的 `meta description` 优先使用 `ai_summary`

**验收标准：** 本地发布一篇文章后，列表页显示 AI 摘要，查看页面源码能看到 meta description 为 AI 摘要内容。

**停机线：**

- 如果 LLM API 响应 >10 秒导致发布体验不可接受 → 停下讨论是否改为异步
- 如果需要引入消息队列或后台任务框架 → 停下确认是否超出 MVP

---

## 第三步：`/sh-run` — 执行

### 你的输入

```
/sh-run

- 当前执行包：AI 摘要生成 MVP 执行包
- 本轮主目标：跑通发布 → 生成摘要 → 展示的完整链路
- 本轮明确不做：摘要自动更新、批量历史补生成、UI 美化
- 验收标准：本地发布文章后列表页显示 AI 摘要，页面源码 meta description 正确
- 哪些情况必须停下来汇报：LLM 调用超时需要改架构、需要引入新的基础设施依赖
```

### Skill 给你的执行报告（摘要）

**完成的工作：** 5 项任务全部完成，摘要生成链路可用。

**难点：** LLM 返回的摘要长度不稳定，有时超过 150 字，需要在 prompt 中加硬约束 + 后端截断兜底。

**权衡：** 摘要生成放在 API 请求同步链路中（发布时多等 2-3 秒 vs 架构简单无需后台任务）。选择同步是因为博客发布频率低（每周几篇），2-3 秒等待可接受，避免引入消息队列的复杂度。

**学习沉淀（踩坑速记）：**

| 踩坑 | 根因 | 解决 | 可复用知识 |
| ------ | ------ | ------ | ----------- |
| LLM 摘要长度不稳定 | 模型不严格遵守字数限制 | prompt 加硬约束 + 后端 `slice(0, 150)` 兜底 | 永远不要只靠 prompt 控制输出长度，代码层必须有截断兜底 |
| Markdown 标记影响摘要质量 | 正文含代码块、链接等标记 | 先 strip Markdown 再送 LLM | 喂给 LLM 的内容要预处理，raw Markdown 会污染生成质量 |

**验收记录（Mode B — 快速确认协议）：**

AI 根据执行报告草拟结论：

```
──验收──
建议：通过
接下来：写个脚本批量补生成历史文章的摘要，或先上线观察效果
  ↳ 主要内容：批量补生成或上线观察
回 ok 确认，有异议直接说
```

## 小结

整个过程中：

- **你决定了**：做什么、不做什么、选哪条路线、验收是否通过
- **Skill 帮你做了**：路线比较、任务拆解、执行实现、难点归因、学习沉淀
- **全程没有出现**：失控的阶段膨胀、无限扩展的文档、被 AI 带着跑的感觉
