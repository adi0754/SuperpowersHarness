# 输出契约

## `sh-start`

按以下顺序返回：

1. 对问题的当前理解
2. MVP 和明确的不做项
3. 路线选项（最多 3 个），每个包含：适用条件、优势、权衡、对 MVP 的影响、对学习价值的影响、何时放弃
4. 收敛部分：推荐路线、理由、不推荐其他路线的关键原因
5. 提示用户填写并确认决策记录：
   - 一句话目标
   - 当前 MVP
   - 明确的不做项
   - 选定路线
   - 不选择其他路线的原因
6. 简洁的主文档仅在决策记录确认后生成，包含：目标、MVP、不做项、路线、最多 3 个结果导向的里程碑、完成标准、人类审批点、最高漂移风险

## `sh-pack`

返回一个当前执行包，包含：

1. 目标成果
2. 为什么现在做
3. 明确的不做项
4. 任务列表
5. 依赖关系
6. 验收标准
7. 停机线
8. 漂移风险

## `sh-run`

返回执行报告，包含：

1. 已完成的工作
2. 未解决的风险
3. 困难点
4. 权衡取舍
5. 需要人类决策的事项
6. 学习摘要 — 使用以下自适应格式：

   **复杂度判定：** 本轮是否涉及 ≥2 次失败尝试、非显而易见的技术决策或重大调试过程？

   **如果否 → 踩坑速记：**

   | 踩坑点 | 根因 | 解决方式 | 可复用知识 |
   |--------|------|----------|-----------|
   | （出了什么问题或让你意外的点） | （为什么会发生） | （怎么修的） | （别人可以直接复用的一句话） |

   如果没有任何问题且任务很直接，可以在"踩坑点"列写"顺利执行"，在"可复用知识"列写关键收获。

   **如果是 → 探索旅程：**

   **问题：**（一句话）

   尝试 N：（你试了什么）
   → 结果：（发生了什么）
   → 洞察：（你学到了什么）

   （对每次重要尝试重复以上格式）

   **可复用知识：**
   - （简洁、可操作的要点，别人可以直接应用）
7. 知识沉淀结果

   执行报告必须明确回答两件事：

   1. **沉淀了什么知识**
      - 只写这轮真正新增、下轮可复用的内容
      - 优先沉淀：踩坑条件、判定标准、操作顺序、验证窍门、停机信号
   2. **沉淀到了哪里**
      - 优先写入项目仓库中的固定知识位置
      - 如果项目还没有固定位置，至少：
        - 在本轮验收文档中增加 `## 知识沉淀`
        - 明确给出文件路径

   推荐最小格式：

   ```markdown
   ## 知识沉淀
   - 落点：{文件路径}
   - 新增知识 1：{一句话}
   - 新增知识 2：{一句话}
   - 下次何时复用：{一句话}
   ```

   护栏：
   - 不要把执行报告全文复制成"知识文档"
   - 不要为了沉淀知识再新开阶段
   - 没有新增知识时，也要明确写"本轮无新增可复用知识，原因是 X"

8. 开发者验收记录 — 根据执行报告草拟推荐答案，然后使用与当前 agent 匹配的交互模式呈现给用户：

   ### Mode A：Claude Code（AskUserQuestion）

   当在 Claude Code 中运行时使用，该环境支持 `AskUserQuestion` 工具。

   **始终必需（至少 2 个问题）：**

   问题 1 — "本轮是否通过？"
   - 选项：AI 根据执行报告草拟 2-3 个选项。始终包含：
     - 推荐结论 + 一行理由（标记为推荐）
     - 如果存在歧义则提供备选结论
   - 选项示例："通过 — 4 项任务全部完成，验收标准已满足"、"有保留 — X 问题待确认"、"不通过 — X 未完成"

   问题 2 — "下一步继续什么？"
   - 选项：AI 根据剩余工作和里程碑草拟 2-3 个下一步选项。
   - 选项示例："进入 Phase 2"、"先上线观察反馈"、"补充 X 再继续"

   **扩展字段 — 当本轮涉及范围变更、意外复杂度或权衡取舍时，作为额外问题包含：**

   - "是否引入了不想要的复杂度？" — 选项："没有"、"有 — （AI 描述具体复杂度）"
   - "本轮需要记录的关键决策？" — 选项：AI 列出报告中的决策，加上"无需额外记录"

   用户选择选项或选择"Other"提供自定义输入。选择后关闭本轮。

   ### Mode B：快速确认协议（Cursor / Codex / 其他）

   当 agent 不支持结构化选择 UI 时使用。

   **用户侧：三行确认提示**

   AI 根据执行报告草拟结论，输出固定格式：

   ```
   ──验收──
   建议：{通过/有保留/不通过}
   接下来：{AI 草拟}
     ↳ 主要内容：{一句话说明具体做什么}
   回 ok 确认，有异议直接说
   ```

   - 用户回复 ok / 确认 / 好 → 关闭验收
   - 用户提出异议 → AI 修正后重新输出同格式提示，不膨胀
   - 修正不限次数，最终 ok 为终态

   **存档侧：自动写入执行报告**

   AI 在输出确认提示的同时，将验收记录追加到执行报告末尾 `## 验收记录`：

   ```markdown
   ## 验收记录
   - 状态：{通过/有保留/不通过}
   - 理由：{一句话}
   - 接下来：{内容}
   - 确认方式：用户确认
   ```

   **扩展字段 — 当本轮涉及范围变更、意外复杂度或权衡取舍时自动追加，不额外询问用户：**

   ```markdown
   - 复杂度评估：{AI 草拟}
   - 关键决策：{AI 草拟}
   ```

   用户修正过的内容记录修正轨迹：`修正：用户将 X 改为 Y`

### `sh-run` 输出违规示例 — 不应产出的内容

**违规 1：没有具体内容的泛泛学习摘要**

> 学习：这一轮很有挑战。我们选了 React 因为它很流行。我们没有考虑其他选项。这在面试中讨论会很好。

为什么错：每个字段都填了但什么也没说。"有挑战"哪方面？"流行"不是技术理由。没有列出实际的替代方案。

**违规 2：对简单任务使用探索旅程**

> 问题：更新配置中的 API 基础 URL。
>
> 尝试 1：改了 URL → 结果：成功了 → 洞察：配置文件很重要。

为什么错：一个没有失败的配置变更不需要探索旅程。使用踩坑速记并填写"顺利执行"即可。

**违规 3：对复杂调试过程跳过探索旅程**

> | 踩坑点 | 根因 | 解决方式 | 可复用知识 |
> |--------|------|----------|-----------|
> | 认证失败 | Token 过期 | 刷新 Token | Token 会过期 |

为什么错：如果调试涉及多次尝试、死胡同和非显而易见的洞察，压缩到一行表格会丢失探索上下文。应使用探索旅程。

## `sh-recover`

返回：

1. 当前主线
2. 什么漂移了
3. 什么应该停止
4. 哪些保持在当前态 vs 执行态 vs 遗留态
5. 文档是否需要重写
6. 下一步推荐的入口
