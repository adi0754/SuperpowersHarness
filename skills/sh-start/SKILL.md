# sh-start — 项目 / 功能启动

本入口驱动：`superpowers:brainstorming` -> 用户决策记录 -> `superpowers:writing-plans`。

## 核心规则

在输入通过检查之前，不得进入工作流。

## 第 1 步：输入检查

本入口使用双字段模型，详见 `references/input-checks.md`。

字段 1（必填）：**你要解决什么？**
- 提示：背景，核心问题，为什么现在做
- 例：企业短视频平台需要声音复刻能力，品牌识别度不足，本季度业务要求上线。

字段 2（AI 推荐）：**什么不能碰？**
- 提示：硬约束，明确不做，已知风险
- 例：合规优先，只做 MVP，优先第三方 API。不做多语言、不做批量导出。

### 置信度分流

- 高置信度：一步确认（AI 填充两个字段，干净展示，无提示）
- 低置信度：先只问「你要解决什么？」，用户回复后 AI 基于回答推荐「什么不能碰？」

## 第 2 步：头脑风暴

使用 `superpowers:brainstorming`。

聚焦于：

1. 这项工作实际解决什么问题
2. 当前 MVP
3. 明确的不做项
4. 关键风险和路线选项
5. 需要人类决定的事项

不要跳入实施或重型 PRD。

### 头脑风暴输出格式

- 最多 3 个路线选项。
- 每条路线必须包含全部六个字段：适用条件、优势、权衡、对 MVP 的影响、对学习价值的影响、何时放弃该路线。
- 以收敛部分结尾：推荐路线、推荐理由、不推荐其他路线的关键原因。

## 第 3 步：决策记录

头脑风暴输出后，提示用户留下决策记录。最低记录必须涵盖：

1. 一句话目标
2. 当前 MVP
3. 明确的不做项
4. 选定路线
5. 不选择其他路线的原因

**在用户留下该记录之前，不得继续生成主文档。**

## 第 4 步：主文档

用户确认决策记录后，切换到 `superpowers:writing-plans` 并编写一份简洁的主文档。

主文档约束：

- 里程碑：最多 3 个。每个必须以结果为导向（例："核心闭环跑通"），而非技术模块导向（例："搭建后端"）。
- 优先说明：为什么存在、目标是什么、明确不做什么、完成标准是什么、哪些点需要人类审批。
- 不要展开为重型 PRD。不要把阶段任务文档作为副产物生成。

## 输出契约

按以下顺序返回：

1. 对问题的当前理解
2. MVP 和明确的不做项
3. 路线选项（最多 3 个），每个包含：适用条件、优势、权衡、对 MVP 的影响、对学习价值的影响、何时放弃
4. 收敛部分：推荐路线、理由、不推荐其他路线的关键原因
5. 提示用户填写并确认决策记录：
   - 一句话目标
   - 当前 MVP
   - 明确的不做项
   - 选定路线
   - 不选择其他路线的原因
6. 简洁的主文档**仅在决策记录确认后**生成，包含：目标、MVP、不做项、路线、最多 3 个结果导向的里程碑、完成标准、人类审批点、最高漂移风险

## 人类控制点

人类必须审批或记录以下决策：

1. 一句话目标
2. MVP 边界
3. 明确的不做项
4. 路线选择
5. 任何范围或结构变更

不得让 Skill 静默吸收这些选择。

## 输出风格

1. 先解释为什么，再解释是什么。以目的引领，而非实施细节。
2. 使用朴素的语言。当朴素描述更清晰时，避免堆叠底层实施术语。
3. 优先为项目负责人的理解而写，其次为执行者的需要。
4. 每份输出必须明确声明：当前目标、为什么现在做、明确不做项、完成标准、需人类审批的点。
5. 不要为了完整性而扩展需求、阶段或文档。

## 护栏

1. 不要为了完整性而扩展项目。
2. 默认不要生成重型 PRD。
3. 除非当前主线确实改变，否则不要创建额外阶段。
4. 在用户留下决策记录之前，不要生成主文档。
5. 优先产出面向项目负责人的内容，而非仅面向执行者。
