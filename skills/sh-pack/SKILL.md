# sh-pack — 构建当前执行包

本入口驱动：`superpowers:writing-plans`。

## 核心规则

在输入通过检查之前，不得进入工作流。

## 第 1 步：输入检查

本入口使用双字段模型，详见 `references/input-checks.md`。

字段 1（必填）：**这轮做到什么程度？**
- 提示：主目标，交付结果，上游依赖
- 例：声音复刻 MVP 核心闭环 — 用户上传授权音频后能生成一条品牌配音并导出。基于 sh-start 拍板记录。

字段 2（AI 推荐）：**这轮什么不碰？**
- 提示：明确不做，跑偏风险
- 例：不做多语言、不做声音风格调整、不做批量导出。API 集成耗时超预期时上报。

### 置信度分流

- 高置信度：一步确认（AI 填充两个字段，干净展示，无提示）
- 低置信度：先只问「这轮做到什么程度？」，用户回复后 AI 基于回答推荐「这轮什么不碰？」

## 第 2 步：编写执行包

使用 `superpowers:writing-plans`。

只构建**一个**当前执行包。

允许的文档类型（只从此列表生成）：

1. 项目主文档
2. 当前里程碑执行包
3. 阶段复盘
4. 异常归因卡
5. 最终项目复盘
6. 面试表达素材包

不要生成此列表之外的文档。不要生成重型 PRD。不要生成多级阶段体系。

## 输出契约

返回一个当前执行包，包含：

1. 目标成果
2. 为什么现在做
3. 明确的不做项
4. 任务列表（4–6 个任务，不要过细）
   - 为什么是 4–6 个：少于 4 通常意味着任务粒度太粗，无法追踪进度或设置停机线；多于 6 通常意味着执行包范围过大，一轮完不成。如果工作确实需要更多，拆分为两个顺序执行包，而非膨胀一个。
   - 过粗示例："搭建后端" — 没有明确的进度信号，没有有意义的停止点。
   - 过细示例："创建用户模型"、"添加 email 字段"、"添加 name 字段"、"添加 email 验证" — 这些是一个任务的子步骤，不是独立任务。
5. 任务间的依赖关系
6. 验收标准
7. 停机线
8. 漂移风险

默认不要扩展出额外阶段或未来执行包。

## 人类控制点

人类必须审批或记录以下决策：

1. 一句话目标
2. MVP 边界
3. 明确的不做项
4. 当前执行包边界
5. 任何范围或结构变更

不得让 Skill 静默吸收这些选择。

## 输出风格

1. 先解释为什么，再解释是什么。以目的引领，而非实施细节。
2. 使用朴素的语言。当朴素描述更清晰时，避免堆叠底层实施术语。
3. 优先为项目负责人的理解而写，其次为执行者的需要。
4. 每份输出必须明确声明：当前目标、为什么现在做、明确不做项、完成标准、需人类审批的点。
5. 不要为了完整性而扩展需求、阶段或文档。

## 护栏

1. 不要为了完整性而扩展项目。
2. 默认不要生成重型 PRD。
3. 除非当前主线确实改变，否则不要创建额外阶段。
4. 优先产出面向项目负责人的内容，而非仅面向执行者。
5. sh-pack 的交付物是执行包文档本身。输出执行包后停止，不要进入 writing-plans 的 execution handoff 步骤。告知用户：「执行包已就绪。准备好执行时，请使用 /sh-run。」
