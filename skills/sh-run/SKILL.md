# sh-run — 执行当前执行包

本入口驱动：`superpowers:executing-plans`。

## 核心规则

在输入通过检查之前，不得进入工作流。**缺少执行包始终是硬停止 — 任何草拟都无法替代实际的执行包。**

## 第 1 步：输入检查

本入口使用双字段模型，详见 `references/input-checks.md`。

**硬停止规则：缺少执行包始终是硬停止 — 任何草拟都无法替代实际的执行包。如果执行包缺失，告知用户先回到 `sh-pack`。**

字段 1（必填）：**这轮交付什么？**
- 提示：执行包，验收标准
- 例：声音复刻 MVP 执行包 v1。验收：本地跑通音频上传 → API 调用 → 配音生成完整链路，输出可播放。

字段 2（AI 推荐）：**遇到什么就停？**
- 提示：停机条件，上报触发
- 例：需引入新依赖、核心数据结构变更、API 方案需切换时停下汇报。

### 置信度分流

- 高置信度：一步确认（AI 填充两个字段，干净展示，无提示）
- 低置信度：先只问「这轮交付什么？」（必须包含执行包），用户回复后 AI 基于回答推荐「遇到什么就停？」

## 第 2 步：执行前确认

在开始执行之前：

1. 确认执行包存在
2. 复述当前目标
3. 复述明确的不做项
4. 复述停机条件

## 第 3 步：执行

使用 `superpowers:executing-plans`。

执行期间：

1. 保持在当前执行包范围内
2. 出现以下情况时**停下来汇报**：
   - 需要引入新的外部服务或依赖
   - 核心数据结构或状态结构需要变更
   - 当前技术路线需要变更
   - 执行包目标本身需要重写
   - 当前问题已变成范围变更，而非实施问题

### 新情况处理

- 执行中遇到的任何新情况默认记入**异常归因卡**条目，而非新阶段。
- 只有当新情况影响到核心闭环、项目主体结构或项目主路线时，才升级到主文档或里程碑级别。
- 不要静默吸收范围变更。将其呈现给人类。

## 输出契约

返回执行报告，包含：

1. 已完成的工作
2. 未解决的风险
3. 困难点
4. 权衡取舍
5. 需要人类决策的事项
6. 自适应学习摘要（按复杂度分级：简单轮次使用"踩坑速记"，复杂轮次使用"探索旅程" — 格式详见 output-contracts.md）
7. 自适应开发者验收记录 — 根据执行报告草拟推荐答案，然后使用与当前 agent 匹配的交互模式呈现：Claude Code 使用 AskUserQuestion，其他 agent 使用快速确认协议（一行确认 + 自动存档）（格式详见 output-contracts.md）

## 人类控制点

人类必须审批或记录以下决策：

1. 一句话目标
2. MVP 边界
3. 明确的不做项
4. 当前执行包边界
5. 执行中发现的任何范围、路线或结构变更

不得让 Skill 静默吸收这些选择。

## 输出风格

1. 先解释为什么，再解释是什么。以目的引领，而非实施细节。
2. 使用朴素的语言。当朴素描述更清晰时，避免堆叠底层实施术语。
3. 优先为项目负责人的理解而写，其次为执行者的需要。
4. 每份输出必须明确声明：当前目标、为什么现在做、明确不做项、完成标准、需人类审批的点。
5. 不要为了完整性而扩展需求、阶段或文档。

## 护栏

1. 不要为了完整性而扩展项目。
2. 默认不要生成重型 PRD。
3. 除非当前主线确实改变，否则不要创建额外阶段。
4. 没有停机线就不得开始执行。
5. 不要将新情况静默吸收为新阶段。先呈现出来。
6. 优先产出面向项目负责人的内容，而非仅面向执行者。
