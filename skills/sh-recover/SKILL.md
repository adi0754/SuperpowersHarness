# sh-recover — 漂移后重新掌控

本入口驱动：`superpowers:brainstorming` -> 可选 `superpowers:writing-plans`。

## 核心规则

在输入通过检查之前，不得进入工作流。

## 第 1 步：输入检查

本入口使用双字段模型，详见 `references/input-checks.md`。

字段 1（必填）：**哪里偏了？**
- 提示：现状，漂移点，手上的证据
- 例：声音复刻 MVP 开发中，执行包范围越来越大，从 MVP 变成了完整语音平台。手上有主文档 v1 和 3 个执行包。

字段 2（AI 推荐）：**想回到什么状态？**
- 提示：恢复优先级，期望产出
- 例：重新确认 MVP 边界，生成一个更小的执行包。

### 置信度分流

- 高置信度：一步确认（AI 填充两个字段，干净展示，无提示）
- 低置信度：先只问「哪里偏了？」，用户回复后 AI 基于回答推荐「想回到什么状态？」

## 第 2 步：阅读当前证据

使用 `superpowers:brainstorming` 重新建立掌控。

先阅读当前证据：

1. 代码和实际产物
2. 当前事实文档
3. 执行文档
4. 仅在需要时查看遗留或存档材料

## 第 3 步：分析漂移

聚焦于：

1. 当前主线到底是什么
2. 什么漂移了
3. 什么应该停止
4. 哪些仍属于当前态 vs 执行态 vs 遗留态
5. 主文档或当前执行包是否需要重写

### 漂移严重程度决策树

使用此决策树确定推荐的恢复动作：

**问题 1：核心目标是否已改变？**
- 是 → **重度漂移** → 推荐 `sh-start`（重新定义方向）
- 否 → 进入问题 2

**问题 2：当前执行包是否仍然有效？**
- 是，但偏离轨道 → **轻度漂移** → 继续 `sh-run` 并修正边界
- 否，执行包范围或任务有误 → 进入问题 3

**问题 3：主文档是否仍然有效？**
- 是 → **中度漂移** → 推荐 `sh-pack`（仅重写执行包）
- 否 → **重度漂移** → 推荐 `sh-start`（重新定义方向）

**严重程度参考：**

| 严重程度 | 信号 | 动作 |
|----------|------|------|
| 轻度 | 超出 1-2 个任务、添加了计划外工作，但目标和执行包完整 | 裁剪回执行包边界，继续 `sh-run` |
| 中度 | 执行包范围有误、任务与现实不符，但主目标成立 | 进入 `sh-pack`，重写执行包 |
| 重度 | 目标本身已偏移、主文档不再反映意图，或路线根本性错误 | 进入 `sh-start`，重新头脑风暴方向 |

## 第 4 步：重写（如需要）

如果需要重写当前事实或执行包，切换到 `superpowers:writing-plans`。

## 输出契约

返回：

1. 当前主线
2. 什么漂移了
3. 什么应该停止
4. 哪些保持在当前态 vs 执行态 vs 遗留态
5. 文档是否需要重写
6. 下一步推荐的入口（`sh-pack`、`sh-run` 或 `sh-start`）

<!-- 共性段落：需在 sh-start、sh-pack、sh-run、sh-recover 之间保持同步 -->
## 人类控制点

人类必须审批或记录以下决策：

1. 一句话目标
2. MVP 边界
3. 明确的不做项
4. 路线选择
5. 任何范围、路线或结构变更

不得让 Skill 静默吸收这些选择。

## 输出风格

1. 先解释为什么，再解释是什么。以目的引领，而非实施细节。
2. 使用朴素的语言。当朴素描述更清晰时，避免堆叠底层实施术语。
3. 优先为项目负责人的理解而写，其次为执行者的需要。
4. 每份输出必须明确声明：当前目标、为什么现在做、明确不做项、完成标准、需人类审批的点。
5. 不要为了完整性而扩展需求、阶段或文档。

## 护栏

1. 不要为了完整性而扩展项目。
2. 默认不要生成重型 PRD。
3. 除非当前主线确实改变，否则不要创建额外阶段。
4. 优先产出面向项目负责人的内容，而非仅面向执行者。
