# Superpowers Harness

**一套面向复杂项目的 superpowers 工作流：让开发者用 AI 提效，而不是被 AI 带着跑。**

> **当项目变大、变新、变复杂时，开发者应该如何使用 superpowers，才能既保持推进效率，又不失去对项目的主导权？**

---

## 这套工作流是怎么来的

越来越多开发者已经在用 AI 做项目开发。

在小需求、局部修改、单点问题修复这类场景里，直接和 AI 对话通常已经足够高效。  
但一旦进入这些场景，问题就会很快出现：

- 一个全新的项目
- 已有项目上的大功能开发
- 复杂需求、多轮规划、多轮执行
- 跨多个模块、多个阶段的推进

在这些情况下，很多开发者都会进入一种熟悉的状态：

- 项目确实在推进
- AI 也确实做了很多事
- **但人对项目的掌控感却越来越弱**

接着就会出现一连串问题：

- 阶段越来越多，文档越来越多
- 提示词越来越长，AI 接管越来越多决策
- 人慢慢只剩下：输入背景、维护上下文、最后验收

最后，项目可能做出来了，  
但开发者自己却不一定说得清：

- 为什么这么设计
- 为什么选这个方案，而不是另一个
- 中途最关键的难点和取舍是什么
- 后面应该怎么维护和迭代

**Superpowers Harness 就是在这种真实开发体验里诞生的。**

---

## 它是什么

Superpowers Harness 不是一个"让 AI 自动把项目做完"的框架，  
也不是一个大而全的软件工程方法论。

它是一套围绕 superpowers 设计的**开发工作流**，目标很明确：

**帮助开发者在新项目、大功能和复杂需求推进中，更轻、更稳、更可控地驱动 superpowers。**

它建立在 superpowers 的三种核心能力之上：

| 能力 | 用于 |
|---|---|
| `brainstorming` | 定锚、澄清、方案比较、异常归因、复盘提炼 |
| `writing-plans` | 生成主文档、执行包、复盘材料等结构化产物 |
| `executing-plans` | 按执行包推进实现，回报难点、权衡和需人工拍板的点 |

所以 Superpowers Harness 的本质不是"增加一套复杂系统"，  
而是：**给 superpowers 增加一套更适合复杂项目的人机协作秩序。**

---

## 它真正解决什么问题

### 问题 1：AI 很容易从"助手"变成"事实上的项目主导者"

它开始替你想方向、拆阶段、生成文档、推进实现、处理异常，  
而你慢慢只剩下输入背景和最后验收。

问题不是 AI 做得不好，  
而是**你的主导权在不知不觉中被流程吃掉了。**

### 问题 2：局部推进很顺，整体主线却越来越模糊

AI 很擅长局部执行。  
但复杂项目的问题，从来不只是"这个局部能不能做对"，更重要的是：

- 当前主线是不是还清楚
- MVP 有没有偷偷膨胀
- 当前问题是实现问题，还是已经变成了范围变化

### 问题 3：项目做完了，但开发者并没有真正"拥有"这个项目

这会带来最现实的后果：说不清项目、难以维护、难以解释架构和取舍、  
难以把项目经验沉淀成自己的能力。

---

**Superpowers Harness 回答的核心问题：**

> 当 AI 已经足够能做事时，开发者怎样才能既享受 AI 的效率，又不失去项目的方向、边界和解释权？

---

## 4 个入口，驱动一次完整的项目周期

`SuperpowersHarness` 把整套工作流压成 4 个入口：

```
/sh-start   /sh-pack   /sh-run   /sh-recover
```

下面用同一个功能——**给短视频平台增加"声音复刻"能力**——演示全部 4 个入口的用法。

---

### `/sh-start` — 启动新项目或大功能

适用场景：新项目刚开始；已有项目要做大功能。

**产出：** 目标收敛 → 路线比较 → 你拍板 → 功能主文档

```
/sh-start 给企业短视频平台增加声音复刻能力
```

或者补充更多背景让结果更准：

```
/sh-start

- 当前项目 / 功能背景：企业短视频平台，已有文案生成、TTS 配音、字幕导出能力
- 这次想解决的问题：增加声音复刻，让客户能用自己的声音生成品牌配音
- 为什么现在做：普通 TTS 品牌识别度低，业务要求本季度上线
- 已知约束：必须控制授权合规风险；只做 MVP；优先接第三方 API
```

Skill 会给你：最多 3 条路线比较（含适用条件、优缺点、对 MVP 的影响）→ 要求你拍板 → 拍板后生成功能主文档。

> 输入太短也没关系——Skill 会先草拟一份模板给你确认，而不是直接开跑。

---

### `/sh-pack` — 把当前目标压成执行包

适用场景：方向已定，要开始一轮执行，需要一个边界清晰的任务包。

**产出：** 任务列表 + 验收标准 + 停机线 + 漂移风险

```
/sh-pack

- 当前主目标：完成声音复刻 MVP 核心闭环
- 当前这一轮要完成的结果：用户上传授权音频后，能生成一条品牌配音并导出
- 当前明确不做：多语言支持、声音风格调整、批量导出
- 依赖的上游结论 / 主文档：sh-start 拍板记录 + 功能主文档 v1
```

Skill 给你一个执行包，只做这一轮，不默认往后展开。

---

### `/sh-run` — 按执行包开始实现

适用场景：已有执行包，开始编码。

**产出：** 实现结果 + 遇到的难点 + 方案权衡 + 需要你拍板的点 + 学习沉淀

```
/sh-run

- 当前执行包：声音复刻 MVP 执行包 v1（或直接粘贴执行包内容）
- 本轮主目标：打通音频上传 → 第三方 API 调用 → 配音文件生成的完整链路
- 本轮明确不做：前端 UI 优化、错误重试机制、批量队列
- 验收标准：本地跑通一条音频生成链路，输出文件可播放
- 哪些情况必须停下来汇报：需要引入新依赖、核心数据结构需要变更
```

执行结束后，Skill 的回报不只是"做完了什么"，还有：

- 本轮最关键的技术难点是什么
- 为什么选了当前方案，放弃了什么方案
- 这轮最值得以后面试讲的一点

最后会返回一个验收记录模板让你填，闭合这一轮。

---

### `/sh-recover` — 项目跑偏后重新接管

适用场景：执行包越来越大、主线越来越模糊、不知道现在在做什么。

**产出：** 当前真相 + 已漂移内容 + 应停止内容 + 下一步建议

```
/sh-recover

- 当前项目 / 功能：声音复刻功能，MVP 开发阶段
- 最混乱或最漂移的地方：执行包已经从"接入第三方 API"变成了"自研声音引擎"，范围翻了 3 倍
- 当前有哪些文档 / 产物 / 代码线索：功能主文档 v1、2 个执行包、当前分支代码
- 这次想先收回什么：重新确认 MVP 边界，砍掉自研部分，生成一个更小的执行包
```

Skill 先读现有证据，告诉你哪些是主线、哪些已经漂移、哪些可以停，再决定是否需要重写执行包。

## 它和一般 AI 开发流程有什么不同

### 它强调 human-in-control

很多 AI 工作流更像"尽可能让 AI 自动规划和执行"。  
Superpowers Harness 更强调：人定目标、人定边界、人定路线，AI 在边界内高效推进。

### 它强调项目主线和收敛能力

不是不断长出更多阶段和文档，而是帮助开发者始终回答清楚：

- 我们现在到底在做什么
- 哪些内容明确不做
- 当前最重要的结果是什么

### 它强调学习沉淀机制

很多 AI 开发方式只强调更快完成、更快生成、更快推进。  
Superpowers Harness 还强调：更好地理解、更好地记录判断、更好地保留项目故事。

这意味着它不仅服务"交付"，也服务：维护、复盘、面试表达、知识迁移、团队传递。

---

## 它适合谁

✅ **适合：**

- 用 AI 做新项目的开发者
- 在已有项目上推进大功能、大需求的开发者
- 不想被复杂提示词和阶段文档拖垮的开发者
- 希望保留项目主导权的开发者
- 希望项目不仅做出来，还能讲清楚、维护得住的开发者

❌ **不适合：**

- 想把整个项目完全交给 AI 自动接管的人
- 只处理极小、极短、单点型需求的场景
- 希望一键生成完整大系统而不做关键判断的人

## 总结

Superpowers Harness 的存在，不是因为 superpowers 不够好。  
恰恰相反，正是因为 superpowers 足够强，复杂项目才更需要一种稳定的使用方式。

如果说直接使用 superpowers 更像是：**把一个问题交给 AI 去推进**

那么 Superpowers Harness 更像是：**让开发者在复杂项目中，学会如何用 superpowers 建立稳定、可控、可复用的推进方式。**

---

**它不是让 AI 接管项目。  
它是让开发者更好地驾驭 superpowers，在复杂开发里既保持速度，也保持掌控。**
